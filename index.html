GRest is set of integration/functional tests bundled together...

it's not a framework, it doesn't do any magic, it is really just bunch of tests.

At the beginning there was manual testing and checklisting

Before each release:

  • prepare what to test and how to do it
  • write it down to confluence
  • check confluence for what was tested in previous releases
  • try it day before release

But we are lazy

We keep forgetting about old tests and we are lazy to test "obvious" things that should work.


If it's cumbersome to test some feature manually, it is more likely to be done wrong.

Duh! That should work. Nobody touched this feature for couple of months, I don't need to test this.


Yeah, sure...

That might be great task for some groovy script

I probably wouldn't write some bigger software in groovy, but for scripting and testing it is sledgehammer

  • great productivity
  • great flexibility
  • no boilerplate code

So we ended up with 400 lines long groovy script.

Automatization was kind of cool but:

  • hard to maintain
  • hard to refactor
  • tests dependencies howto ?
  • logging howto?
  • if one test fail, it shouldn't affect other tests
  • ...
Use the right tool for the right job!
Don't reinvent the wheel!

GRest was born

  • standard java/groovy project in msf repo
  • built on top of gradle (maven + ant + groovy)
  • built on top of TestNG
  • asserts, before/after, tests isolation
  • configurable standardized logging
  • nice test reports
  • some basic reusable pieces of code
  • composition, inheritance

We have tests for

  • some resource tests
  • aka process, schedule, metadata storage - create/delete/update
  • some cloud connect graph tests
  • facebook, salesforce, google analytics, exact target, dataset integration, ...
  • end 2 trash tests for testing single cc component
  • end 2 end tests for testing integration - ged data from Salesforce and upload it to GoodData
  • scheduler run graphs
  • ...

There are multiple test groups

  • SLA - tests triggered every couple of minutes to check that msf services are up and running
  • HEALTHCHECK - thorough tests primary for checking msf services after release
  • REST - tests triggered as commit hook (former PERL tests)
  • DOWNLOADERS - workers tests with mocked backend

Every test(except Downloaders) can be triggered against any server

  • respective jenkins job, check MSF view
  • manually from devel machine or PI (check README for more info)


Note that tests in DOWNLOADERS group need target machine with custom setup (because of mocking 3rd party backends like facebook, salesforce,...)

Implementation

Groovy - use the right tool for the job

  • fast implementation
  • concise - no boilerplate code
  • very productive
  • handy closures
  • string interpolation
  • collections literals
  • operator overloading
  • ...

Gradle

  • good project to try out gradle
  • easy customization
  • groovy support
  • concise syntax
  • gradle wrapper
  • incremental build
  • dependencies are bundled together when deploying on PI (to avoid downloading them)

TestNG

  • very similar to JUnit
  • test groups
  • tests dependencies
  • easy parallelism setup
  • data providers

HttpBuilder

  • easy to use but hard to customize
  • TT refreshing hell
  • no IDE help (thanks to dynamic nature and lack of typing)


Will be replaced with RestAssured and basic authentication

jadler

For backend mocking in downloaders test

Some restrictions and known bugs

  • environment has to be set manually (datasets, eventstores)
  • issues with HttpBuilder - ie. TT renewal doesn't work at all (RestAssured will solve this with basic auth)
  • improve logging (verbose, debug)
  • test inheritance versus composition - reusing common code (msf backend?)
  • Simple adding new graph tests (using TestNG data provider)
